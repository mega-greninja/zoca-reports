03 — BUILDING BLOCKS
ZOCA

Three concepts. Everything else is wiring.

💬

Conversation

One row per merchant + customer. Created on first contact. Never splits. If closed and customer returns — same row reopens.

Key = merchantId + customerPhone
📎

Unit (Event)

One row per thing that happened. Inbound call, outbound SMS, booking, merchant note — each is a unit, stored in order.

📞 inbound-call 📞 outbound-call 💬 inbound-sms 💬 outbound-sms 📋 lead-form ✅ booking 👤 merchant-action
🧠

Classifier

One LLM prompt. One job. Reads last 10 units → outputs a 5-dimension state object. Runs after every new event. Never sends anything.

Input: 10 units → Output: state JSON

3 existing pieces (already in production)

💾 chatbot.followups table

One row per scheduled outbound. Has clientId, scheduledTime, mode (SMS/PHONE), message, processed status.

⚙️ WinFollowupProcessor

Bull worker. Picks up when scheduledTime arrives. Sends via Twilio (SMS) or voice provider (call).

🔧 request-followup tool

The AI agent calls this to schedule outbound. It's the bridge between AI decisions and actual delivery.

🛑 DND — 2 checkpoints

Check 1: When followup row is created (reject if DND)
Check 2: Right before sending (abort if DND set since)

07 — AUTO-CLOSE
ZOCA

Two things happen without any new event

Bull delayed jobs handle time-based state transitions. No LLM needed — rules are deterministic.

24h

Cold Close

After the last outbound, if no inbound arrives within 24 hours → auto-close as Lost (cold)

↺ Resets on each new outbound
✕ Cancels if any inbound arrives
7 days

Action Item TTL

If a non-pinned action item sits unresolved for 7 days → auto-close as Lost (cold)

📌 address-complaint is PINNED
→ never auto-closes. Only manual Close.

⚡ Hard rule: STOP keyword in latest SMS → immediately Closed, Lost(dnd) regardless of anything else in flight.

10 — API SURFACE
ZOCA

6 reads. 5 writes. 4 queues.

📖 Read endpoints

GET /inbox/entity/:id — Conversation list (paginated, filterable)

GET /inbox/conversations/:id — Header + state + action items

GET /inbox/messages/:unitId — Unit drill-down (transcript, tools)

GET /frontdesk/activity/entity/:id — AI activity feed

✏️ Write endpoints

POST .../messages — Send manual SMS (via follow-up)

POST .../close — Force close → Handled

POST .../spam — Mark as spam

POST .../notes — Add note (no classifier rerun)

PATCH .../read — Toggle read/unread

11 — ROLLOUT & SCOPE
ZOCA

Opt-in rollout. Clear boundaries.

✅ How it rolls out

🚦 Feature flag per entity

Every webhook branches: inbox_v2 = true → new path. Non-enrolled entities see zero changes.

🗄️ Separate tables

V2 writes to V2 tables only. Existing tables untouched. No retroactive migration.

🧪 Test mode

isTest = true hides from production. Visible with ?includeTest=true.

🖥️ Frontend gate

Inbox + Activity tabs only mount for enrolled entities.

Websockets / live streaming

Per-action-item resolve buttons
