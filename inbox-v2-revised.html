02 — BUILDING BLOCKS
ZOCA

Three concepts. Everything else is wiring.

💬

Conversation

One row per merchant + customer. Created on first contact. Never splits. If closed and customer returns — same row reopens.

Key = entityId + customerPhone
📎

Unit (Event)

One row per thing that happened. Call, SMS, booking, note — each is a unit in chronological order.

📞 inbound-call 💬 inbound-sms 📋 lead-form ✅ booking 👤 merchant-action
🧠

Classifier

One LLM prompt. One job. Reads last 10 units → outputs 5D state. Runs after every event. Never sends anything.

10 units in → 5D state JSON out

07 — GAP #2: RETELL WEBHOOKS
ZOCA

The original PRD only mentioned call_ended

But Retell sends 3 webhooks. The analysis arrives 5-30s after the call ends.

📞

call_started

call_id, agent_id, from/to number, start_timestamp

→ Cache in Redis (TTL 2h)
🔚

call_ended

duration_ms, disconnect_reason. No transcript. No analysis.

→ Cache end data. Do NOT create unit yet.
🧠

call_analyzed

Summary, transcript, tool_calls, sentiment, bucket — everything the classifier needs.

→ Create unit + run classifier HERE

Edge case: call_analyzed never arrives

Bull delayed job at call_ended + 60s. Creates minimal unit with duration only. Classifier runs with limited data. If call_analyzed arrives later → update unit + re-classify.

Where in the codebase

libs/win/src/lib/services/
frontdesk-retell-webhook.service.ts
Handles all 3 webhook types already.

08 — OUTBOUND
ZOCA

Classifier labels. Follow-up system sends.

All outbound goes through existing infrastructure. Nothing to rebuild.

SMS reply flow
1. Customer SMS arrives (Twilio webhook)
↓
2. Save unit + enqueue classifier (runs in parallel with AI agent)
↓
3. AI agent decides reply → calls request-followup → DND check #1
↓
4. WinFollowupProcessor sends → DND check #2 → Twilio → delivery webhook
↓
5. Save outbound-sms unit → classifier re-runs with updated timeline
3 existing pieces (all in production)

💾 chatbot.followups

scheduledTime, mode (SMS/PHONE), type (RESPONSE/REMINDER), trigger (BOT/CALL/SMS), processed, error.

⚙️ WinFollowupProcessor

Bull worker, concurrency 20. Job ID: win_followup_{id}. Re-fetches row before processing.

🔧 request-followup MCP tool

AI agent calls this to schedule outbound. Writes to chatbot.followups. Available in FrontdeskService.

Two independent LLM calls: AI reply agent ("what to say?") + Classifier ("what state?"). They don't interact.

09 — GAP #3: DND (4 CHECKS)
ZOCA

DND has 4 checkpoints, not 2

The original PRD mentioned 2. The real system has 4 enforcement layers plus auto-lift.

1
request-followup tool
Check l2b.do_not_disturb.isActive. Reject if true — followup row never created.
↓
2
WinFollowupProcessor (before send)
Re-check DND. If set since #1 (e.g., customer just texted STOP) → abort + deactivate followup.
↓
3
TwilioService.sendSms()
Check twilio.phone_blocklist. Hard block — these numbers are permanently blocked, never auto-lifted.
↓
4
Classifier input (dnd_on_file flag)
Classifier sees DND status → immediately sets Lost(dnd). Prevents stale UI showing "Active" for a DND'd customer.

Auto-lift: When a DND'd customer calls back → DoNotDisturbRepository.disable() fires. isActive → false. Conversation reopens. The MCP tool enable-do-not-disturb also cancels all pending followups.

12 — GAP #5: ERROR HANDLING
ZOCA

What happens when things fail

The original PRD had zero error handling. Here's the retry strategy.

🧠 Classifier failures

LLM timeout (>15s)
Retry 2x (5s, 10s backoff). If all fail → Sentry alert, state unchanged, unit.state_at_time = null.
Invalid JSON response
Retry 1x. If still invalid → log error, leave state unchanged.
Out-of-schema enum values
Validate against allowlists. Reject + retry 1x.

🔒 Infrastructure failures

Redis lock timeout
Retry 3x (1s backoff). If all fail → requeue job with 30s delay.
Optimistic lock conflict
Silent no-op. A newer classifier already wrote. Expected behavior.
call_analyzed never arrives
60s delayed job creates minimal unit. Re-classify when data arrives.

📊 Observability

Classifier latency
Prometheus histogram per run.
Error rate
Sentry + Prometheus counter on failures.
Queue depth
Bull metrics for all 3 new queues.
Cost tracking
LLM token usage per entity per day.
